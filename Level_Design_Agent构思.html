[PART 01]

核心判断与原则

Agent 并非万能。能用简单 Flow 解决的问题，不优先上 Agent；能用单 Agent 解决的问题，不轻易拆成多 Agent。

[CORE INSIGHT]

关卡设计管线的核心矛盾在于信息在人与人之间传递时的失真、流程编排的低效、以及品质标准的不统一，而非简单的"缺少智能"。

真实痛点 本质 适合的方案
信息在人与人之间传递时失真 格式化 + 结构化问题 SKILL TOOL
白盒→美术→程序的交接成本高 流程编排问题 FLOW + 局部 AGENT
大世界模块多，依赖关系复杂 任务拆解与追踪问题 FLOW + PLANNER AGENT
设计意图模糊，迭代时上下文丢失 知识管理 + 语义理解问题 AGENT + MEMORY
品质标准不统一，review 靠经验 评估标准化问题 SKILL + AGENT

渐进式路径

Phase 1: Skill 层 当前阶段

把确定性的事情先自动化——标准化文档生成、格式转换、数据提取。不需要 Agent，不需要决策能力。

Phase 2: Flow 层

把管线流程编排起来——任务拆解、依赖管理、自动调度。用 DAG 编排器串联多个 Skill。

Phase 3: Agent 层

在关键决策节点引入智能判断——设计意图理解、跨模块协调、品质审查。

Phase 4: 范式探索

重新思考"关卡策划"这个角色的工作方式——从文档驱动到对话驱动，从线性生产到生成-评估循环。

  • 视觉参考

中间表示 (IR) 设计

[KEY DESIGN DECISION]

所有输入（图片、对话、文档）都在 InputProcessor 阶段被转化为统一的结构化 IR。下游 Skill 永远不接触原始输入——图片变成文字描述，对话变成决策摘要。节省 token 的真正解法在于路由，而非压缩。

{ "level_id": "string", "level_name": "艺术馆潜行", "type": "POI | MainMission | SideQuest", "SPACE": { "regions": [{ "id", "name", "function", "connections", "spatial_notes" }], "critical_path": ["A", "B", "D", "F"], "optional_paths": ["A", "C", "E", "F"] }, "FLOW": { "nodes", "edges", "critical_path" }, "FEEL": { "emotional_beats", "visual_references", "audio_intent" }, "MECHANIC": { "mechanics", "ai_behaviors", "parameters" }, "ASSET": { "required_assets", "reuse_candidates" }, "SYSTEM": { "dependencies", "interfaces", "constraints" }, "uncertainty_flags": [{ "field", "note": "策划未明确，AI推断" }], "source_refs": [{ "type", "path", "summary" }] }

[PART 05]

上下文治理体系

大体量多 Skill 开发的核心挑战：如何在开发第10个 Skill 时，仍然保持高效生成和整体一致性？

问题的三个层面

[开发]

开发时

第10个 Skill 开发时，前9个的 Schema/Prompt/约束已经记不清了，容易冲突。
→ 开发态的知识管理

[运行]

运行时

多个 Skill 串行执行时，上下文窗口装不下所有信息。
→ 执行态的上下文路由

[迭代]

迭代时

改了一个 Schema 字段，不知道会影响哪些下游 Skill。
→ 变更态的影响追踪

[COUNTER-INTUITIVE PRINCIPLE]

上下文窗口不需要"看到全貌"，全局一致性依靠局部自治、契约约束和自动化校验来实现。全局一致性 = 局部自治 + 契约约束 + 自动化校验。这和微服务架构的思想一致——每个服务不需要知道整个系统，只需要遵守接口契约。

三层治理架构

LAYER 1

Contract Layer（契约层）

一切的基础。定义 IR Schema、Skill 注册表、渲染规范、全局约束。

  • IR Schema（6维度，唯一真相源）
  • Skill Registry（所有 Skill 的接口契约 — "电话簿"）
  • 术语表、命名规范、渲染标准
  • 存储为独立文件，纳入版本控制
  • 使用者：开发者 + 自动化校验工具
LAYER 2

Dev Context Layer（开发上下文层）

每个 Skill 的开发只需要看到与自己相关的最小信息集。

  • 自动生成的 Skill Dev Brief（~3-4K token，恒定大小）
  • 包含：自己的输入/输出 Schema、上下游列表、项目约束、术语表
  • 不包含：其他 Skill 的内部实现、IR 中不相关的维度
  • 使用者：开发 Skill 的你（或 Claude Code）
LAYER 3

Runtime Context Layer（运行上下文层）

每个 Skill 执行时只注入最小必要上下文。

  • System Prompt（固定，~1K token）
  • IR 子集（按契约声明裁剪，~1-2K token）
  • 上游产物（仅依赖项，~0.5-1K token）
  • 示例（预算允许时，~1K token）
  • 使用者：运行中的 LLM

契约层详细设计

契约层是整个系统的"宪法"——所有 Skill 的开发和运行都必须遵守。

文件结构

/contracts/ ├── ir_schema.json # 6维度 IR 的完整 JSON Schema ├── skill_registry.yaml # 所有 Skill 的注册表（"电话簿"） ├── glossary.yaml # 术语表 ├── render_standards.yaml # 渲染规范 ├── constraints.yaml # 全局约束 │ ├── skills/ │ ├── bubble_chart/ │ │ ├── contract.yaml # 接口契约 │ │ ├── prompt.md # System Prompt │ │ ├── examples/ # 输入输出示例 │ │ └── tests/ # 测试用例 │ ├── flow_chart/ │ │ └── ... │ └── .../ │ └── validators/ ├── schema_validator.py # IR 合法性校验 ├── contract_validator.py # Skill 输出合规校验 └── consistency_checker.py # 跨 Skill 一致性校验

Skill Registry（技能注册表）

任何时候想了解全局，看这一个文件就够。每个 Skill 只声明接口，不包含实现。

skill_registry.yaml
registry_version: "1.0.0" skills: BubbleChart: version: "1.0.0" purpose: "生成玩法流程气泡图" ir_dimensions_read: [FLOW, FEEL] ir_dimensions_written: [] upstream: [InputProcessor] downstream: [POIDocAssembler, FlowChart] render_format: "SVG via Mermaid" status: "development" AssetList: version: "1.0.0" purpose: "生成需求物件表" ir_dimensions_read: [ASSET, SPACE] upstream: [InputProcessor] downstream: [POIDocAssembler, ArtHandoff] render_format: "Excel via openpyxl" status: "planned"

单个 Skill 的 Contract

skills/bubble_chart/contract.yaml
skill_name: BubbleChart version: "1.0.0" # === 输入契约 === input: ir_fields_required: - path: "FLOW.nodes" - path: "FLOW.edges" - path: "FLOW.critical_path" - path: "FEEL.emotional_beats" # === 输出契约 === output: format: "mermaid_flowchart" constraints: - "必须包含 START 和 END 节点" - "所有 IR.FLOW.nodes 中的节点必须出现" - "不得添加 IR 中不存在的节点" # === 质量契约 === quality: auto_checks: - name: "node_coverage" rule: "输出节点集 == 输入节点集" severity: "error" - name: "mermaid_syntax_valid" rule: "Mermaid CLI 能成功渲染" severity: "error"

Dev Brief 自动生成

[CORE MECHANISM]

开发任何 Skill 时，运行 generate_dev_brief.py SkillName，自动生成只包含"与我相关"的最小上下文。无论开发第1个还是第20个 Skill，上下文大小恒定（~3-4K token）。

Dev Brief 包含的内容
# 自动生成的 Skill 开发简报 ## 1. 你要开发的 Skill → 名称、职责、输出格式 ## 2. 你的输入（IR 中你能看到的字段） → 只展示契约声明的 IR 子集 Schema ## 3. 你的上游（谁给你数据） → 上游 Skill 的输出格式（不含实现） ## 4. 你的下游（谁消费你的输出） → 下游 Skill 对你的输出有什么期望 ## 5. 项目约束 + 术语表 → 只提取与本 Skill 相关的部分 ## 6. 已有类似 Skill 的模式参考 → 保持风格一致性 ## 7. 你不需要关心的事情 → 明确边界，防止越界

自动化一致性校验

在三个时机自动运行，确保全局一致性：

[开发]

开发时校验

改了任何契约文件后自动运行。检查：上下游引用合法性、关系对称性、IR 维度合法性、无循环依赖。

[变更]

变更时校验

Skill 契约变更后运行。检查：输出字段是否向后兼容、被删除字段是否有下游依赖、版本号是否更新。

[运行]

运行时校验

每次 Skill 执行后运行。检查：输出是否符合契约 Schema、自动质量检查是否通过、与输入的一致性。

评分与迭代机制

两种打分结合：自动即时评分决定是否重试，人工反馈评分驱动 Skill 迭代。

自动评分（每次执行后立即运行）

维度检查内容权重
结构完整性必填字段存在、流程图有完整路径、节点数合理30%
一致性气泡图节点与流程图对应、Layout区域与IR匹配30%
规范符合度命名规范、格式模板、[待确认] 标记20%
信息保真度LLM 对比 IR 和产物，检查信息丢失或编造20%

人工评分（异步，使用产物后填写）

[KEY: KEEP IT SIMPLE]

不要让策划填10个维度的表。只问3个问题：
可用性："拿到这个文档，我能直接开始工作吗？" (1-5)
准确性："文档描述的和我理解的设计意图一致吗？" (1-5)
完整性："有没有我觉得重要但文档遗漏的信息？" (1-5)

评分驱动的行为

[重试]

自动评分 < 0.6

自动重试（最多2次），使用不同的生成策略。

[失败]

自动评分 < 0.4

标记为失败，通知开发者排查。

[迭代]

人工评分持续 < 3

触发 Skill 迭代 review，分析低分原因。

[PART 08]

范式转变展望

Phase 1-2 是优化现有管线。真正的范式变化在于重新定义"关卡策划的工作方式"。

[对话]

转变 1：文档驱动 → 对话驱动

Agent 成为设计意图的"活的载体"。美术无需翻文档，直接向 Agent 提问："这个大厅的氛围应该是什么感觉？" Agent 带着溯源地回答。

[循环]

转变 2：线性生产 → 生成-评估循环

画完 Layout 后，Agent 即时生成动线热力图、视线分析、节奏曲线预测。迭代周期从周/月缩短到小时/天。

[协作]

转变 3：模块化分工 → 语义化协作

CrossModuleCoordinator Agent 监控各模块状态，自动识别跨模块一致性问题。如"策划设计了关灯潜行，但AI系统未实现光照影响警觉度"。

[FOUNDATION FOR THE FUTURE]

结构化 IR Schema → Agent 的"思考格式"
评分数据 → Agent 的"奖励信号"
执行日志 → Agent 的"经验库"

[PART 09]

风险与边界

风险等级风险说明应对
高 Memory Pollution Agent 记住过时的设计决策 版本化知识库，标注信息来源和时间
高 Hallucinated Actions Agent 编造策划未做过的设计决策 所有输出可溯源，无法溯源标注 [推断]
中 Context Window 溢出 大世界关卡信息量巨大 分层摘要 + 按需检索，三层上下文治理
中 过度依赖 团队依赖 Agent 而非自己思考设计 Agent 定位为"信息助手"，禁止做创意建议
低 Prompt Injection 恶意或无意输入导致异常 输入校验 + 输出审查 + 权限隔离

[CORE OBJECTIVE]

AI Agent 对关卡生产管线的最大价值在于成为团队的"共享大脑"——让每个人都能即时访问完整的设计上下文，消除信息在人与人之间传递时的损耗。先把 Skill 层做扎实（标准化），再让 Agent 成为团队的上下文层（语义化协作），最后才考虑更深的自动化。

[APPENDIX]

日常 Skill 开发工作流

标准化的开发流程，确保每个 Skill 的开发体验一致。

Step 1: 注册

在 skill_registry.yaml 中注册新 Skill（声明输入输出维度、上下游关系）

Step 2: 校验

运行 consistency_checker（确认注册信息与现有 Skill 无冲突）

Step 3: 生成 Dev Brief

运行 generate_dev_brief.py SkillName（自动生成开发简报，~3-4K token）

Step 4: 开发

把 Dev Brief 交给 Claude Code，生成 contract.yaml 和 prompt.md

Step 5: 测试

编写 2-3 个测试用例，运行测试检查输出是否符合契约

Step 6: 评审

人工评审输出质量，记录评分

Step 7: 迭代

迭代 Prompt 直到测试通过 + 评分达标，提交并更新 registry status

[CHANGE RULES]

任何时候修改已有 Skill 的契约：① 必须先运行 check_contract_compatibility ② 必须更新版本号 ③ 必须在 change_log 中记录原因和关联的低分案例
