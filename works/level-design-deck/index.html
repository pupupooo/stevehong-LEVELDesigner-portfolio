WHY IT MATTERS

这套工具想加速的，不只是写文档这一个动作。

开放世界关卡策划经常承接大量落地沟通：玩法、空间、美术、灯光、音频、VFX、程序、系统都要从同一份设计中读取不同信息。传统文档即使写得很详细，也很难保证每个职能都能马上找到自己开始工作所需的上下文。

降低文档事务负担

设计师不应该把大量时间消耗在重复整理格式、拆分汇报材料、复制字段、维护多份文档一致性上。设计源事实稳定后，文档和 Deck 应该从同一份结构化记录派生出来。

spec.json 真源 schema-driven 编辑 自动渲染

降低沟通摩擦

真正耗时的不是一次写作，而是职能之间反复确认“你到底需要我做什么”“我现在还缺什么信息”。职能接口文件可以提前暴露缺口，把模糊需求转成可追问、可审核、可交付的材料。

职能接口文件 缺口追问 交付前检查
INTERFACE EXAMPLES

接口文件不是新模板，而是让协作对象更快开始工作的契约。

下面两个样例都是暂定接口边界，用来说明产品机制。成熟形态中，每个接口都应由实际使用它的职能参与校准。

EXAMPLE 01

玩法原型接口：GD 到 TD/GPP 标准文档

当前流程通常是 GD 写玩法需求，TD 根据需求实现玩法原型组件，GD 再组装调试；原型过审后再转正式需求，由 GPP 开发正式功能。每一层都可能发生重新解释和反复沟通。

TD/GPP 共同维护长期更新的项目专属蓝图 MCP，沉淀可复用蓝图组件、实现模式、项目约束和接口要求。
GD 在 design deck 中完成玩法设计后，工具根据蓝图 MCP 的接口要求生成 GD 到 TD/GPP 标准文档。
实现同事不再从宽泛设计文档中反推需求，而是审核标准文档，监管 AI Agent 调用项目专属蓝图 MCP，并在关键节点确认实现逻辑。
纯原型阶段可以进一步压缩链路：GD 调用经过验证的蓝图能力完成可审查原型，再由接口层转译为 GPP 需要的正式开发信息。
项目专属蓝图 MCP GD -> TD/GPP 关键逻辑审核
EXAMPLE 02

灯光接口文件（暂定示例）

这个示例不是要替灯光师定义最终规范，而是说明：当关卡策划填写设计源事实时，工具可以根据“灯光开始工作需要什么信息”提前检查缺口。

已有源事实

区域功能、情绪曲线、时间段、关键视线、危险/安全区域、重点事件节点。

开始工作可能需要

主视觉焦点、导航引导、明暗节奏、动态变化条件、美术和 TA 依赖。

工具应追问设计师

哪些区域需要灯光引导？哪些区域必须压迫或迷失？关键事件是否需要状态切换？

必须由灯光/TA 校准

实时光和烘焙光边界、性能预算、后处理依赖、材质响应和平台限制。

WebApp 最新交互演示

这段演示记录了当前 WebApp 的一次交互迭代：气泡图视图已经可以和结构化表单联动，点击节点后自动定位到对应字段，并支持在同一工作台里完成查看、编辑、保存与重新渲染。

Web UI 把告警栏、schema 表单、HTML 预览和 Chat 放到同一工作台里，设计师不必在命令行、编辑器和浏览器之间来回切换。
表单由 schema 驱动生成，字段存在与否回到结构定义，而不是在 HTML 里自由发挥。
机械检查优先发现占位、断链、类型越界和跨模块引用问题，AI 自信不能替代硬约束。
同一份设计源事实可以派生为汇报 Deck、长文档和后续接口材料。

PRODUCT FORM

最终形态不是聊天框，也不是文档编辑器，而是面向设计师的对话优先工作台。

我正在参考 `nexu-io/open-design` 这类交互形式：设计师从 IM 式对话进入，工具先澄清问题、给出计划、生成可预览产物，再把关键变化放回人工批准门。今天的 WebApp 已经把 Chat 放进工作台，下一步是让 Chat 更深地写入 spec、调用检查与生成能力。

产品形态展望

对话只是入口，真正的产品能力在于把对话里的意图转成可维护的源事实，并用项目知识、职能接口和机械检查不断约束它。

  • 对话优先交互：用追问和计划降低设计师进入成本。
  • 结构化编辑：关键字段回到 schema 和源事实，而不是散落在长文本里。
  • 可视化检查：把区域关系、流程节点、派生产物和缺口报告放在可审查界面中。
  • 人工批准门：体验判断、关键逻辑和跨职能接口变更必须由人确认。
  • 项目目标对齐层：注入项目目标、约束、模式和参考案例，让新设计不从空白 prompt 开始。
  • 项目专属蓝图 MCP：让经过维护的实现能力进入原型链路，同时保留 TD/GPP 的维护和审核权。
