开放世界沙盒任务系统

一个以"生动自由"为核心体验的 2D 沙盒游戏系统设计。
灵感来源：Schedule I 的涌现式世界 × 科恩兄弟电影的荒诞叙事。

01设计目标与灵感来源

目标

创造一个"围绕玩家展开"的随机叙事开放世界体验——事件会在玩家附近自行发展、意外收场或反高潮结束；玩家是这个世界的主角。

利用反情节等手段去创造 "一个真正的开放世界沙盒"

传统开放世界任务的问题：所有事件都为玩家精心编排，NPC 在路边等着被拯救，强盗会配合地被打败，宝箱永远有好东西。他们像任务清单一样等着玩家，沉浸感崩塌。

Anti-Plot（反情节）借鉴了科恩兄弟、北野武式的叙事手法——最强力的杀手可能在浴室滑倒摔死；精心策划的抢劫被一只流浪猫打断；等了半天的决战对手压根没来。

尝试用"不按套路出牌"的体验，使用系统驱动叙事，让系统规则自然产生意外。

灵感参照：《Schedule I》与《羊蹄山之魂》

玩《Schedule I 》的时候，被开头两处连续的反情节体验所吸引，进而联想我们能否在大世界探索中创造这样的体验去替代那些曾经绕不开的清单化支线任务。同时从《羊蹄山之魂》的卡牌遭遇系统中获得了更多设计上的启发

出发点：大世界载具玩法与撒点机制的思考

在完成载具玩法的框架设计后，在讨论撒点方案时，意识到当前设计工具和思考框架的薄弱与不足，尝试引入对马岛团队所提出的卡牌概念。在反复的思考和尝试后得出导演+卡牌的设计框架

02核心问题

已有非清单化做法的困境

  • 纯随机和目标驱动 → 缺乏戏剧张力
  • 目标导向行动计划 → 自运行世界与玩家无关
  • NPC底层自治 → 从下至上的体验生成，难控制
  • 高度定制化 → 生产成本过高

MVP尝试提供一套解法

  • 模板化：8-12 种 Beat 模板覆盖大部分戏剧结构
  • 槽位化：运行时由导演系统填充具体角色
  • 修饰器：环境/状态驱动的变异规则
  • 归因系统：追踪"谁做了什么"，产生连锁后果

03Beat 三层架构

Beat（节拍）是系统的基本叙事单元。每个 Beat 不是一段固定剧本，而是一个可被填充和变异的戏剧模板。

┌─── Beat 三层结构 ───────────────────────────────────────┐ │ │ │ Layer 1: 模板 (Template) │ │ 定义"戏剧结构"，如 [对峙] [请求] [伏击] [发现] │ │ 约 8-12 种模板覆盖所有常见戏剧冲突 │ │ │ │ │ ▼ │ │ Layer 2: 槽位填充 (Slots) │ │ 导演系统在运行时选择具体实体填入角色槽位 │ │ 同一个 [对峙] 模板：[守卫 vs 小偷] 是正剧 │ │ [乞丐 vs 乞丐] 是荒诞剧 │ │ │ │ │ ▼ │ │ Layer 3: 修饰器 (Modifiers) │ │ 根据环境/玩家状态注入"变异规则" │ │ [对峙] + [实力悬殊] = 弱方直接投降（反高潮） │ │ [对峙] + [截胡] = 野猪冲进来搅局（荒诞） │ │ │ └──────────────────────────────────────────────────────────┘

Beat 模板配表结构

BeatTemplate:
  id: "confrontation"          # 对峙
  slots:
    - role: "aggressor"            # 攻击方
      tag_filter: [hostile, greedy]
      min_count: 1
      max_count: 3                 # 防止选入太多演员
      proximity_radius: 350       # 候选最大距离
    - role: "victim"
      tag_filter: [civilian, merchant]
      min_count: 1
  exit_conditions:
    - condition: "victim.hp <= 0"     → "victim_dead"
    - condition: "aggressor.hp <= 0"  → "aggressor_dead"
    - condition: "elapsed > 120"      → "timeout"  # 安全超时
  modifiers: [hijack, power_gap, weather_slip]

修饰器冲突解决

当多个修饰器同时满足条件时，按 priority 排序，取优先级最高者。同类修饰器通过 exclusive_group 互斥。

关键原则：宽进严出

  • 触发条件尽量宽松 —— 只要有两拨人就可以触发 [对峙]
  • 退出跃迁才做细分 —— 根据谁死了、怎么死的、玩家参与度决定走向
  • 这样保证 Beat 高频触发，同时结局多样

04反情节模式工具箱

7 种可复用的反情节模式，每种对应一类修饰器：

#模式核心体验系统依赖情感标签
1 截胡型 Hijack 问题自行解决，玩家白来了 实体自治 AI + 环境物理 荒诞感拔剑四顾心茫然
2 沉没成本型 Sunk Cost 投入时间/资源，纸面回报为零，但通过标签传递，意外之处给予回报 时间系统 + 行为追踪 + token传递 被戏耍提高社区讨论度
3 蝴蝶效应型 Butterfly 微小行为引发连锁灾难 Token 传递 + 阵营联动 失控黑色幽默
4 立场反转型 Betrayal 帮助的人反过来伤害你 NPC 隐藏标签 好心没好报世界冷漠感
5 过期变质型 Decay 延迟行动导致任务状态变异 生命周期计时器 + 实体演化 世界不等你错过的遗憾
6 无声尴尬型 Awkward Silence 所有铺垫之后什么都没发生 导演"反高潮"调度 反高潮虚惊一场
7 旁观者效应型 Bystander 你的不作为也有后果 感知追踪 + 旁观者记录 道德拷问不作为的代价

频率控制：二八法则 + 动态权重

  • ~80% 正常：Beat 按玩家预期执行（杀怪完成任务、给苹果得感谢）
  • ~20% 反情节：修饰器激活，产生意外结局
  • 渐进调节：前 30 分钟反情节比例降至 5-10%
  • 熔断器：连续 2 次反情节后强制 3 个正常 Beat
  • 同类冷却：同一种反情节模式 10 分钟内不重复

05导演系统准则

导演系统是整个反情节体验的"总调度"。以下 6 条准则构成了它的设计规范——任何具体实现都不能违反这些原则。

准则 1：只制造情境，不强制行为

导演可以在玩家前方刷出一群追着 NPC 跑的野猪。导演不可以强制玩家去救那个 NPC、不可以锁定镜头/操作。玩家永远有"转头走人"的自由——走人本身也是一个有效的退出节点。

准则 2：惩罚来自世界规则，不来自导演意志

好的反情节：玩家违法 → 警察系统检测到 → 追捕（因果链）。
坏的反情节：玩家太安逸 → 导演直接删掉玩家的车（天降惩罚）。
即使是"掀桌子 Beat"，也必须有合理的世界内因果。

准则 3：导演的存在感必须为零

实体被选为"演员"后，行为过渡必须平滑。一头正在觅食的野猪被选角后，应该看起来是"自然地注意到了食物味道"。没有"事件开始"横幅。玩家应该是事后回想才意识到"那一连串事情是连起来的"。

准则 4：给玩家搅局的权力

每个 Beat 必须支持"暴力打断"。玩家在对峙 Beat 演到一半时扔炸弹炸了所有人？→ 必须有 Abort 分支。打断不是"失败"，而是另一种有趣的结局。三级打断：微干扰（旁观）→ 直接干预（攻击参与者）→ 掀桌子（AOE）。

准则 5：频率的二八法则

80% 正常 + 20% 反情节。如果每次都反转，反转就变成新的"套路"。导演系统有调度冷却计时器 + 连续熔断机制。详见上方"频率控制"。

准则 6：公平承诺

当反情节给玩家造成损失时，系统在后续提供非对称补偿机会——不是直接退还损失，而是开启一条独特路径（隐藏区域、NPC 的同情赠礼），让"遭遇不幸"变成独特体验。

★从 MVP 中提炼的三个核心设计原则

以下三条原则是在原型反复迭代中被验证的。我希望通过这些原则让系统能产生有意义的涌现体验。

原则 ①：以玩家为中心的导演系统

对比：全局调度 vs 玩家中心扫描

传统方式（全局调度）

给每个 NPC 一个目标系统（GOAP / Utility AI），让它们自行决策。世界到处在"演"，大部分精彩事件发生在玩家看不见的地方。
结果：一个自运行的世界，与玩家无关。

我的方式（玩家中心扫描）

导演系统只在玩家附近扫描候选实体，只在玩家可感知范围内触发 Beat。没有观众的表演没有意义。
结果：每一次事件玩家都能参与/目睹，以玩家体验为中心。

这意味着：

  • 森林另一头的山贼和商人再近，只要不在玩家视野内，导演不会在那里安排表演
  • 玩家走到哪里，关卡与任务的"舞台"就搭到哪里——这是一个跟随摄影机的剧组
  • NPC 在"离场"时仍有自治 AI（闲逛/觅食），保持世界呼吸感，但不消耗导演资源
  • 导演在世界稳定运转后才开始工作——先让世界的舞台搭起来，再安排演出

原则 ②：宽进严出的 Beat 触发

"触发条件尽量宽松，退出跃迁才做细分。"

想要避免的情景：精心设计的事件因为触发条件太苛刻而永远不发生。

宽进：触发一个 [对峙] Beat 只需要两个条件： ① 有一个 hostile 标签的实体在玩家附近 ② 有一个 civilian 标签的实体在玩家附近 就够了。不检查等级、不检查阵营关系、不需要任务前置。 严出：同一个 [对峙] 的结局可以是 8 种以上： ├── aggressor_dead ← 玩家/守卫击杀山贼 ├── victim_dead ← 商人被杀（玩家未干预） ├── hijack ← 野猪冲进来截胡 ├── power_gap ← 山贼感受到威压逃跑 ├── weather_slip ← 山贼在雨里滑倒 ├── player_intimidation ← 玩家举枪威慑 ├── player_dead ← 玩家自己倒下了 └── timeout ← 超时自然消散

期望目标：

  • 高频触发：宽松的入口确保玩家几乎每走一段路就能遇到事件，不会有"跑图空窗期"
  • 多样性来自出口而非入口：增加一个新结局只需加一个修饰器，不需要新建 Beat 模板
  • 内容杠杆高：4 个 Beat 模板 × 8 个修饰器 = 理论上数十种独特体验组合
  • Beat 有类型优先级：位置触发型（如洞穴发现）优先于人物触发型（如对峙），确保稀缺体验不被常见事件抢占

原则 ③：结构化自顶向下的 Beat 设计

"模板 → 槽位 → 修饰器 → 退出条件"，每一层只负责一件事。

这条原则的目标是让项目能"收住"——涌现任务系统的一大风险是复杂度失控。四层结构让每一层有明确的职责边界，新增内容只在对应层操作，不会波及其他层。

Layer 1: 模板（决定了戏剧的类型） 4 种模板覆盖了对抗/探索/委托/护送 每种只定义戏剧结构，不定义具体角色或结局 │ ▼ Layer 2: 槽位（决定了演员） 导演系统在运行时选择演员 同一个 [护送] 模板：老王去驿站 vs 小陈去驿站 = 不同的路线和风险 │ ▼ Layer 3: 修饰器（决定了剧本的走向） 8 个修饰器，各自独立，互不耦合 立即触发型：截胡、实力悬殊、天气意外 ← Beat 创建时就生效 延迟触发型：找包裹反转、商人反水 ← Beat 进行中持续检测世界状态 │ ▼ Layer 4: 退出条件（减少混沌，收束可能） 每个模板定义所有可能的退出节点 Beat 引擎持续监测，任一条件满足立刻收场

每一层的职责边界：

层谁负责输入输出
模板设计师（配表）戏剧类型槽位定义 + 修饰器白名单
槽位导演系统（运行时）Tag 过滤条件具体实体绑定
修饰器Beat 引擎（条件检测）世界状态快照应用/不应用
退出条件Beat 引擎（持续监测）实体生死/位置/超时结算 + 归因

扩展验证：从MVP设计时第一版（2 Beat + 4 修饰器）扩展到第三版（4 Beat + 8 修饰器），核心架构零改动。新增的 [护送] Beat 和 [商人反水] 修饰器完全复用了已有的 Beat 生命周期。这证明了四层结构的可扩展性——增加内容不会增加架构复杂度。

来自《羊蹄山之魂》动态遭遇卡牌系统的启发

参考来源

Sucker Punch 在 GDC 上分享的「Dynamic Encounters Deck System」——一套用于开放世界骑马旅途中随机遭遇的"抽卡/牌组引擎"。它用极低的性能开销替代了全图实体模拟，通过导演后台"发牌"来控制玩家遇到什么。

核心理念

《羊蹄山》的做法是：世界并不在远方自行运转，导演手里的牌决定了玩家随机遭遇的事件"。随着玩家骑马移动，导演按节奏抽牌，抽到什么牌就在玩家前方的高速缓存区刷什么事件。牌组里有大量"空牌"来保证节奏张力。

这套系统本质上解决的是"长距离移动中的事件填充"问题——让骑马赶路不无聊。

尝试在框架设计上进行借鉴与优化

框架上的学习与优化：Beat 导演系统本质也是"后台打牌"，尝试对照他们的做法，迭代导演系统：

关键判断

不需要推翻重来。羊蹄山的 Deck System 解决的是"骑马赶路不无聊"这个单一问题，而我们的导演系统在 Beat 三层架构、归因结算、多阶段生命周期、记忆系统上已经很完整了。

借鉴的方式：从中提取 3 个具体机制，作为增量补丁打到现有导演系统上。

导演系统进化：3 个增量机制

机制 1：注入队列已实现

对应羊蹄山的"塞牌"：让世界事件不仅广播给周围 NPC，还能往导演的待触发队列里注入高优先级 Beat 条目。

旧流程 玩家偷苹果 → emitWorldEvent('crime') → 周围NPC反应 → 结束 新流程 玩家偷苹果 → emitWorldEvent('crime') → 周围NPC反应 ↘ injectBeat('wanted_confrontation', {priority: HIGH, ttl: 300s}) ↓ 导演下次扫描 → 先检查注入队列 → 找到 'wanted_confrontation' → 在玩家前方刷出守卫拦截

机制 2：紧张度控制已实现

对应羊蹄山的"空牌比例控制"：让 world.tension 值真正参与 Beat 触发/抑制决策，而不只是被动衰减。

紧张度控制beats逻辑 导演扫描触发 → 读取 world.tension │ ├── tension > 70（高紧张）→ 60% 概率跳过本轮（等效"抽到空牌"） │ 玩家刚经历激烈战斗，需要喘息，导演"按住不发" │ ├── tension 30~70（正常）→ 正常流程 │ └── tension < 30（低紧张）→ 提升高冲突 Beat 的触发权重 玩家跑图太久没事发生，导演主动制造冲突

效果：高紧张时自动冷却，低紧张时主动加压。相比羊蹄山"移除所有高冲突牌到弃牌区"更简单，效果一致。

机制 3：加权抽取设计完成 / 待实现

对应羊蹄山的"洗牌"：将 tryTriggerBeat() 从"固定优先级链式结构改为基于上下文的加权随机选取。

旧做法：固定优先级 tryTriggerBeat(): if (tryDiscovery()) return; ← 永远第一 if (tryConfrontation()) return; ← 永远第二 if (tryEscort()) return; ← 永远第三 tryRequest(); ← 永远第四 新做法：加权随机，设计方案 tryTriggerBeat(): candidates = [ ← 收集所有可触发的 Beat { beat: 'confrontation', weight: base × tensionMultiplier }, { beat: 'escort', weight: base × cooldownBonus }, { beat: 'request', weight: base × reputationFactor }, ] selected = weightedRandom(candidates) execute(selected)

06系统架构

完整版分为 5 个模块。MVP 阶段每个模块只实现最小可验证子集。

Module 1: World Foundation (世界基础) ├── TileMap（平地、悬崖、建筑内） ├── Weather System（☀️晴天 🌧️雨天 ⛈️雷暴） ├── Day/Night Cycle └── Global Variables Blackboard Module 2: Entity System (实体系统) ├── Base Entity（HP, Position, Faction, Tags[]） ├── Goal-Driven AI（觅食、巡逻、逃跑） ├── Sensor System（感知范围、天气、其他实体） ├── Damage Source Tracker（归因核心） └── Actor Mode Switch（自治 ⇄ 演员模式） Module 3: Director System (导演系统) ├── Player Proximity Scanner ├── Tension Graph（紧张度曲线） ├── Beat Selector（选择 Beat + 演员） ├── Cast Manager（槽位填充） └── Anti-Plot Cooldown Timer Module 4: Beat Engine (节拍引擎) ├── Beat Template Loader ├── Modifier Evaluator（修饰器评估） ├── Exit Condition Monitor ├── Credit Evaluator（归因结算） ├── Token Manager（蝴蝶效应） └── Abort Handler（打断处理） Module 5: Feedback Layer (反馈层) ├── Dynamic Dialogue（归因驱动对话） ├── World Event Broadcast（世界事件广播） ├── Simple UI └── Event Log（决策日志）

07MVP 范围

MVP 核心假设

"同一个模板 + 不同修饰器 = 不同的情节体验"

只需验证这一个假设。不需要完整 RPG 系统、背包、战斗数值、多区域地图。

08当前 H5 原型概览

原型采用模块化 JS 架构（10 个文件），零依赖，双击 index.html 即可运行。

模块结构

index.html     # 入口：HTML/CSS + 输入/游戏循环胶水代码 (~350 行)
js/config.js   # 常量：地图区域、颜色、建筑、树木
js/utils.js    # 纯函数：距离、钳位、随机、几何
js/world.js    # 全局状态：实体管理、日志、玩家死亡/复活
js/events.js   # 世界事件广播 + NPC 分类型反应
js/ai.js       # NPC AI 状态机（idle/wander/chase/flee/attack/patrol）
js/beats.js    # Beat 模板 + 修饰器 + setup/active/resolve 生命周期 (~400 行)
js/director.js # 导演扫描 + 优先级触发 + Beat 选择
js/gun.js      # 枪械：瞄准/射击/装弹/射线检测
js/render.js   # Canvas 渲染：世界/实体/UI/特效 (~220 行)
js/debug.js    # 调试面板：参数滑块 + 手动触发 + 状态监控

原型包含的系统

世界

  • 6 个功能区域：森林、集市、卫兵营、道路、洞穴、驿站
  • 天气系统：Q 键切换晴/雨
  • 粒子雨效果影响修饰器条件

实体

  • 8 个实体（含玩家）
  • 状态机 AI：idle / wander / chase / flee / attack / patrol
  • 伤害来源追踪 + 记忆系统
  • 玩家 HP / 死亡 / 复活机制

导演（玩家中心扫描）

  • 定时扫描（可调）+ 启动缓冲 + 冷却系统
  • 4 Beat 模板 + 8 修饰器
  • Beat 类型优先级：发现 → 对峙 → 护送 → 请求
  • 反情节冷却 + 同类去重

玩家能力

  • 枪械系统：左键瞄准/威慑，右键射击，R 装弹
  • 近战攻击 + 上下文交互（接受/拒绝任务、拾取物品）
  • 世界事件广播 → 周围 NPC 语境反应
  • 实时事件日志（导演决策全透明）

🛡️ 护送🌧️ 天气意外截胡型(天气)下雨立即
🗡️ 商人反水立场反转型40% 随机 + 护送途中 2 个山贼 < 200px延迟

延迟触发修饰器：

传统修饰器在 Beat 创建时就决定结局。但最好的反情节体验来自"一切进行顺利...然后突然反转"。

[立场反转]：玩家帮旅人找回包裹，交还那一刻——旅人发现守卫在旁边，立刻大喊"就是这个人偷了我的包裹！"
[商人反水]：护送商人走到一半，路过两个山贼——商人突然翻脸，和山贼一起攻击玩家。

延迟触发的条件在 updateBeatActive() 中持续检测，而不是在 setup 阶段一次性判定。

09NPC 图鉴

当前原型包含 7 个 NPC，每个都有独立的 AI 行为、世界事件反应规则、以及上下文感知对话。

🧑‍🌾

商人老王

civilianmerchant

驻区：集市 | HP: 60 | 战力: 2 | 速度: 35

AI 行为：在集市内缓慢踱步。遇到战斗立刻逃跑。

对玩家行为的反应：

  • 玩家杀山贼 → "太感谢你了！" (记入记忆，后续见面引用)
  • 玩家杀平民 → "杀人了！！" 然后逃跑
  • 玩家高声望靠近 → "欢迎欢迎！大英雄来了！"
  • 玩家低声望靠近 → "别...别过来..." (退缩)
  • 附近发生战斗 → 逃离战斗区域
🧳

货郎小陈

civilianmerchant

驻区：集市 | HP: 50 | 战力: 1 | 速度: 30

AI 行为：与老王类似，活动范围略有不同，提供额外的 Beat 受害者候选。

设计用途：确保导演系统有足够的 civilian 槽位候选人。两个商人意味着即使一个在 Beat 中，另一个仍可被选为下一个 Beat 的演员。

💂

守卫阿强

civilianguardauthority

驻区：卫兵营 (巡逻至集市) | HP: 150 | 战力: 12 | 速度: 55

AI 行为：在卫兵营和集市之间巡逻。发现暴力事件会前去调查/追捕。

对玩家行为的反应：

  • 玩家杀山贼 → "干得好，公民！" (正面)
  • 玩家杀平民 → "住手！你被捕了！" 然后追捕玩家 (最重要的世界规则反馈)
  • 看到山贼攻击平民 → 追捕山贼
  • 玩家低声望靠近 → "你看起来很可疑..."

设计角色：守卫是"世界规则"的执行者，体现准则 2——惩罚来自世界规则。

🗡️

山贼大刘

hostilegreedy

驻区：森林 | HP: 80 | 战力: 8 | 速度: 50

AI 行为：在森林游荡。被导演选为 aggressor 时向目标移动并攻击。

对玩家行为的反应：

  • 同伴被玩家杀 + 玩家较弱 → "你会后悔的！" 主动攻击玩家
  • 同伴被玩家杀 + 玩家较强 → "我不想死！" 逃跑
  • 记忆中玩家危险 + 玩家靠近 → 自动回避

设计角色：Beat [对峙] 的主要 敌人 候选。两个山贼确保系统有持续的冲突源。

⚔️

山贼二毛

hostilegreedy

驻区：森林 | HP: 70 | 战力: 6 | 速度: 45

AI 行为：与大刘类似但稍弱。当大刘被消灭后，二毛会根据记忆选择报仇或逃避。

设计用途：提供第二个 hostile 候选，确保单次击杀不会清空所有威胁源（另一个在冷却/重生期间系统仍能触发对峙）。

🐗

野猪

animalhungry

驻区：森林 | HP: 90 | 战力: 15 | 速度: 40

AI 行为：在森林缓慢觅食游荡。被惊吓时快速逃跑。

对玩家行为的反应：

  • 附近发生任何战斗/击杀 → 受惊逃跑
  • 高声望/危险玩家靠近 → 逃跑

设计角色：[截胡] 修饰器的核心执行者。当它在受害者附近时，导演系统可以让它"自然地"冲向山贼，截胡整个对峙。

🚶

旅人阿花

civiliantraveler

驻区：道路 | HP: 40 | 战力: 1 | 速度: 35

AI 行为：沿道路来回行走，是世界"活力感"的填充者。

对玩家行为的反应：

  • 玩家杀山贼 → "好厉害！这位侠士真勇猛！"
  • 玩家杀平民 → "杀人犯！" 然后逃跑
  • 记忆中有目击事件 → "我听说你杀了[某某]..."
  • 附近战斗 → 立刻逃跑

设计角色：作为"路人甲"提供世界氛围。也是潜在的 Beat 受害者候选。后续版本可作为 [请求] Beat 的委托人。

10世界事件系统

世界事件系统是连接"玩家行为"和"NPC 反应"的桥梁。它解决的核心问题是：玩家的行为不应该在真空中发生。

工作原理

玩家行为 ──→ emitWorldEvent(type, data) ──→ 半径内所有实体 │ ▼ ┌─────────────────────┐ │ 对每个实体执行: │ │ reactToWorldEvent() │ │ 根据实体类型分发: │ │ → merchantReaction() │ │ → guardReaction() │ │ → banditReaction() │ │ → boarReaction() │ │ → travelerReaction() │ └─────────────────────┘

事件类型

事件触发时机广播半径携带数据
kill任何实体死亡400px凶手、受害者、位置
combat_nearby战斗发生300px攻击者、目标、是否涉及平民
gunshot玩家开枪500px射击者、位置
intimidate玩家持续瞄准 NPC200px施压者、目标
crime_witnessed修饰器触发（如旅人栽赃）400px嫌疑人、证人

NPC 反应矩阵

事件商人守卫山贼野猪旅人
玩家杀山贼感谢赞许仇恨/恐惧逃跑崇拜
玩家杀野猪赞叹中性忌惮—赞叹
玩家杀平民逃跑追捕无反应逃跑逃跑
山贼攻击平民逃跑追捕山贼—逃跑逃跑
枪声逃跑巡逻调查*忌惮/逃跑逃跑逃跑
瞄准威慑退缩警告恐惧/逃跑逃跑退缩
犯罪被发现逃跑立即追捕——逃跑

* 守卫听到枪声后进入 patrol 状态前往枪声位置调查；只有玩家声望 < 25 时才直接追捕。

记忆系统

每个 NPC 有 memory[] 数组（上限 10 条），记录目睹的事件。记忆影响后续对话和行为：

  • 商人记得你杀过山贼 → 下次见面："多谢你之前出手相助！"
  • 山贼记得你杀过同伴 → 远距离就开始回避你
  • 旅人记得目击事件 → "我听说你杀了[某人]..."

手动触发

  • 触发 [对峙]：强制创建一个对峙 Beat（忽略冷却）
