[GLOBAL VIEW]

三级权限模型

[READ]

read — 读取

读取 IR 子集、查询 Bible、检索模式。
自动执行，不需确认，不留审计日志。

[PROPOSE]

propose — 提案

提交 IR 变更提案、生成文档产物。
自动执行，但所有提案留日志，变更需审核后合并。

[EXECUTE]

execute — 执行

调用 UE 工具、修改引擎资产。
必须人类确认。执行前快照，可回滚。Phase 3+ 才开放。

capability_registry.yaml (权限声明)
permission_model: levels: read: auto: true audit: false propose: auto: true audit: true # 所有提案留日志 execute: auto: false # 必须人类确认 audit: true checkpoint: true # 执行前快照
[PART 03]

多Agent通信协议

灵感来源："从零开始"文章中的子Agent隔离（Git worktree 分支模型）与显式消息传递。核心原则——Agent 间不直接通信，所有消息经过总指挥路由。

IR 分支模型

[DESIGN PATTERN: BRANCH & MERGE]

每个职能Agent不直接修改主IR，而是在 IR 的"分支"上操作——提交变更提案，由总指挥（或人类）审核后合并。所有变更显式、可审计、可回滚。

ir_branch_proposal.yaml（变更提案示例）
# 美术Agent对 stealth_museum_01 的IR修改提案 ir_branch: source: "stealth_museum_01/ir/v003_merged.json" author: "art_agent" changes: - path: "SPACE.regions[2].spatial_notes" old: "大厅，20m x 15m" new: "大厅，25m x 18m" reason: "原尺寸放不下策划要求的4个掩体+巡逻路线" - path: "ASSET.required_assets" action: "append" value: { id: "pillar_gothic_02", count: 6 } reason: "扩大后需要额外柱子分割视线" side_effects: - "FLOW.edges 中 A→B 的距离可能需要重新计算" - "FEEL.emotional_beats[0] 的节奏可能因空间变大而被稀释" status: "pending_review" # pending → approved → merged / rejected
!

为什么不直接修改

"信息在人与人之间传递时失真"——Agent和Agent之间同样存在这个问题。分支模型让所有变更都是显式的、可审计的、可回滚的。每个提案必须声明 reason（可溯源）和 side_effects（跨维度影响预判）。

三种消息类型

借鉴"从零开始"文章中的最小化多Agent协调协议——"没有消息队列，没有共享状态，只有显式的消息传递。"

[MSG-1]

task_assignment

总指挥 → 职能Agent

分发任务：指定 POI、操作类型（generate / review / revise）、IR子集范围、Bible预检索摘要。

[MSG-2]

change_proposal

职能Agent → 总指挥

提交变更：IR分支 + 产物列表 + Agent自评置信度。总指挥审核后决定合并或拒绝。

[MSG-3]

coordination_request

职能Agent → 总指挥 → 另一职能Agent

跨职能协调：必须经过总指挥路由。声明问题、提出的变更、向对方提的问题。

message_protocol.yaml（协议定义）
message_protocol: version: "1.0.0" routing_rule: "Agent间不直接通信，所有消息经过总指挥路由" types: task_assignment: from: "orchestrator" to: "<functional_agent>" payload: poi_id: "string" action: "generate | review | revise" ir_subset: "IR维度列表" bible_hints: "编排器预检索的Bible摘要" change_proposal: from: "<functional_agent>" to: "orchestrator" payload: ir_branch: "IR分支结构" artifacts: "产物列表" confidence: "0.0 - 1.0" coordination_request: from: "<functional_agent>" via: "orchestrator" # 必须路由 to: "<another_functional_agent>" payload: issue: "string" proposed_change: "IR变更摘要" question: "string"
[WHY CENTRALIZED ROUTING]

总指挥作为中心节点可以：① 检查消息格式合规 ② 过滤不相关上下文（上下文路由）③ 记录所有通信日志（可审计）④ 在必要时升级给人类决策。防止 Agent 间点对点通信制造信息失真。

[PART 04]

当前设计回退风险审计

逐项扫描关卡Agent方案中的设计决策，标注在大循环下哪些可平滑演进、哪些需要预留接口、哪些必须现在改。

低风险：可平滑演进

当前设计大循环下的演进路径为什么低风险
6维度 IR Schema作为所有Agent的通信协议，不变IR 是"关卡是什么"的建模，与产出形式无关
上下文三层管理从"管理Skill开发"扩展到"管理多Agent协作"契约层的思想直接适用于 Agent Registry
评分机制从"评文档"扩展到"评引擎产物"框架不变，评分维度加字段即可
Bible双层架构不变，甚至更重要飞轮机制与产物形式无关
渐进式路径（Phase 1→4）节奏正确，重新定义每Phase产物即可策略层面的决策，架构层面无负债

中风险：需要预留接口

当前设计潜在问题预防措施
Skill Registry 只注册 Skill 大循环还需注册 Agent、Tool（UE侧）、通信协议 改名为 capability_registry.yaml，预留 agents: / tools: namespace
IR 的 source_refs 只引用设计输入 大循环下IR需要双向引用——不仅"从哪来"，还要"去了哪" 加 downstream_refs 字段（可为空，但字段要有）
contract.yaml 的 render_format 绑定文档格式 render_format: "SVG via Mermaid" 暗示输出必然是文档 改为 output_type，值域：document / engine_action / data_export

⚠️ 高风险：必须现在改

[HIGH RISK: ORCHESTRATOR COUPLING]

当前编排器把"上下文路由"和"执行调度"耦合在一起。在大循环下，这两个职责归属不同的层：上下文路由→每个Agent自己的契约声明；执行调度→总指挥Agent的决策。必须现在解耦。

[当前：路由和调度耦合]

orchestrator:
  pipeline:
    - skill: BubbleChart
      inject: [FLOW, FEEL]  ← 耦合

[改后：路由在契约，编排器只调度]

# contract.yaml 声明需要什么
input:
  ir_fields_required: [FLOW, FEEL]

# 编排器只做两件事：
# 1. 读contract → 裁剪IR → 组装
# 2. 按依赖关系调度执行

[WHY THIS MATTERS]

这个改动很小（重构2-4小时），但它让编排器从"路由+调度一体机"变成"纯调度器"。未来总指挥Agent接管调度时，上下文路由的逻辑完全不用改。

[PART 05]

知识库结构扩展

当前 Bible 架构方向正确，但有两个结构性问题会在多Agent写入和模式规模化时暴露。现在改动成本极低。

Level Bible 按维度 / 职能分区

[PROBLEM]

当前设计："一个关卡一个Bible，内部扁平"。多Agent时代，关卡Agent和美术Agent同时写同一个 ir_current.json——谁的版本是对的？design_rationale.md 是策划的意图还是美术的调整理由？

level_bibles/ 改进后的目录结构
level_bibles/ stealth_museum_01/ ir/ current.json # 当前合并后的 IR（真相） history/ v001_level_design.json # 关卡Agent写入 v002_art_feedback.json # 美术Agent反馈的修改 v003_merged.json # 合并版本 rationale/ level_design.md # 关卡策划的设计意图 art_direction.md # 美术方向的解释 tech_constraints.md # 程序的技术约束 scoring/ document_scores.json # 文档产物评分（Phase 1） engine_scores.json # 引擎产物评分（Phase 3+，现在为空） human_feedback.json # 人工评分

这个目录结构不影响 Phase 1 的任何逻辑——当前只有关卡Agent在写入。但它避免了未来多Agent写同一个Bible时的合并地狱。

模式条目索引结构

[PROBLEM]

当前 Project Bible 的模式库没有定义检索索引。从5个模式增长到500个时，"检索"不能靠全文扫描。

pattern_library/stealth_corridor.yaml（带索引）
metadata: id: "pattern_stealth_corridor_01" tags: [stealth, corridor, linear, indoor] ir_dimensions: [SPACE, FLOW, FEEL] # 涉及哪些维度 applicable_poi_types: [stealth, infiltration] confidence: 0.82 # 置信度 citation_count: 7 # 被引用次数 last_validated: "2025-03-01" # 上次验证 content: pattern: "本项目潜行段落通常采用线性主轴 + 1-2个可选分支..." anti_pattern: "避免单图超过12个节点..." examples: ["stealth_museum_01", "stealth_lab_03"]
[SCALABILITY]

有了索引字段，检索用纯字段匹配+排序：filter(poi_type) ∩ filter(ir_dimensions) ∩ sort(confidence × citation_count)。500个模式检索延迟在毫秒级，不需要 embedding，不需要向量数据库。等真的需要语义检索时，这些索引字段也是向量化的好起点。

改动 2：编排器解耦

把上下文路由规则从编排器内部移到 contract.yaml 中声明。编排器变成"纯调度器"——读取契约、裁剪IR、组装上下文、按依赖顺序执行。

!

为什么这是最重要的改动

未来总指挥Agent接管调度时，它会替换编排器的调度逻辑（从静态DAG变为Agent Loop）。但上下文路由是稳定的——它由契约声明驱动，不随调度方式改变。解耦后，路由逻辑零改动。

[PART 07]

实施优先级排序

七项改动按优先级排序。总额外工作量约 1-2 天，换来 Phase 1 所有核心资产能无痛过渡到大循环。

优先级改动工作量价值
P0 Registry 升级：skill_registry → capability_registry，预留 agents / tools namespace，Skill 加 owner_agent 字段 30 分钟 避免未来整体重构注册表
P0 编排器解耦：上下文路由规则提取到 contract.yaml，编排器只做"读取+调度" 2-4 小时 避免最大的回退风险

P1 Level Bible 分区：ir/ rationale/ scoring/ 子目录结构 1 小时 避免多Agent写入冲突
P1 Pattern 索引 metadata：每个模式条目加 tags / ir_dimensions / confidence / citation_count 30 分钟/条目 支持规模化检索
P2 协议文档：写 message_protocol.yaml 和 permission_model.yaml 2-3 小时 给大循环留蓝图
P2 UE 工具接口契约：在 capability_registry 的 tools: 下定义 planned 状态的接口 1-2 小时 提前对齐 UE 程序员

阶段映射：从验证器到完整形态

Phase 1：关卡Agent（DAG） 当前

目标：证明"设计师对Agent说清楚意图 → IR → 产物"的循环可行
编排：静态 DAG，够用
产物：文档/图表（output_type: document）
同时做：7项改动，为大循环预留接口
验证标准：用3-5个真实POI跑通端到端流程

Phase 2：多职能Agent + 总指挥

目标：Agent间协作跑通
编排：总指挥运行 Agent Loop，职能Agent提交变更提案
新增：美术Agent、IR分支合并机制
产物：仍以文档为主，但包含跨职能交接包
验证标准：关卡Agent和美术Agent协作完成一个POI设计

Phase 3：UE 工具层接入

目标：从"文档产出"到"引擎产出"
新增：UE MCP Server、execute权限开放、引擎内验证
产物：output_type: engine_action 开始出现
关键：UE工具粒度定义、执行反馈回传机制
验证标准：Agent调用UE工具放置一个简单POI的物件

Phase 4：闭环 + 飞轮加速

目标：完整循环——意图 → IR → 引擎产物 → 验证 → Bible → 下一个关卡更快
新增：NPC AI Agent、程序Agent、跨关卡质量检测
飞轮：品质模型校准完成，系统自动拦截低质量产出
验证标准：第N+1个关卡的设计时间比第N个显著缩短

[CORE CONCLUSION]

Phase 1 的真正目标不是"生成16种文档"，而是证明"意图 → IR → 产物 → 验证"的循环可行。文档只是当前阶段的产物形式。如果循环成立，未来换成 UE 工具调用，只是替换 Tool Layer——上面的所有东西（IR、Registry、Bible、评分、上下文路由）全部可以复用。
